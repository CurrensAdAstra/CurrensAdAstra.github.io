# GPU 교통 시뮬레이션이 어려운 이유

도시 교통 시뮬레이션은 계산량이 매우 큰 대표적 문제입니다.  
현실적인 도시 규모 시뮬레이션은 복잡한 도로 네트워크에서 상호작용하는 수천~수백만 대의 차량을 모델링해야 합니다. 그래서 직관적으로는 GPU가 완벽한 플랫폼처럼 보입니다.

GPU는 대규모 병렬성, 높은 메모리 대역폭, 데이터 병렬 워크로드에 대한 효율적인 실행 특성을 제공합니다. 이론상으로 교통 시뮬레이션은 GPU에서 매우 잘 확장될 것 같습니다.

하지만 실제로는 **GPU 기반 교통 시뮬레이션은 놀랄 만큼 어렵습니다**.

이 글에서는 효율적인 GPU 교통 시뮬레이터 구현이 왜 처음 생각보다 훨씬 어려운지 핵심 이유들을 정리합니다.

---

## 직관: “차량을 그냥 병렬화하면 되지 않을까?”

처음 떠올리기 쉬운 아이디어는 단순합니다.

> 각 차량은 독립적이다 → 차량당 GPU 스레드 하나를 배정한다.

개념적으로는 다음과 같습니다.
> 차량 1 → 스레드 1
> 차량 2 → 스레드 2
> 차량 3 → 스레드 3
> ...
> 차량 N → 스레드 N

겉보기에는 GPU 실행 모델과 아주 잘 맞습니다.

문제는 교통 시스템이 **독립적이지 않다**는 점입니다.  
각 차량의 행동은 주변 차량 상태에 강하게 의존합니다.

예를 들어:

- 선행 차량 추종(car-following)
- 차선 변경
- 교차로 충돌 회피
- 신호등 반응

이런 상호작용은 시뮬레이션 엔티티 간 강한 의존성을 만듭니다.

---

## 문제 1: 차량 간 데이터 의존성

교통 모델은 대개 **국소 상호작용(local interaction)** 에 의존합니다.

예시:
> 차량 B의 속도 ← 차량 A의 위치

차량 A가 감속하면 차량 B는 즉시 반응해야 합니다.

CPU에서는 순차 실행으로 비교적 쉽게 구현되지만,  
GPU에서는 수천 개 스레드가 동시에 실행됩니다.

이때 다음 문제가 발생합니다.

- 차량 상태의 비일관 읽기
- 경쟁 상태(race condition)
- 동기화 오버헤드

모델이 현실적일수록 의존성 처리 복잡도는 더 커집니다.

---

## 문제 2: 불규칙 메모리 접근

GPU는 메모리 접근 패턴이 예측 가능하고 정렬되어 있을 때 가장 효율적입니다.

교통 시뮬레이션은 보통 도로망을 표현하는 **그래프 구조** 위에서 동작합니다.

예시:
> 도로 세그먼트
> ↓
> 차선
> ↓
> 차량 목록

실제 접근 패턴은 다음을 자주 포함합니다.

- 연결 리스트
- 인접 그래프
- 희소 공간 구조

이 구조들은 **불규칙 메모리 접근**을 유발하며, GPU에 불리합니다.

스레드가 흩어진 주소를 읽으면 메모리 트랜잭션 병합(coalescing) 효율이 떨어지고, 결과적으로 전체 처리량이 감소합니다.

---

## 문제 3: 부하 불균형

교통 시뮬레이션에서는 모든 도로가 동일하게 혼잡하지 않습니다.

예시:
> 고속도로 구간 → 200대
> 주거지 도로 → 5대

각 도로 구간에 동일한 계산 자원을 배정하면 많은 스레드가 유휴 상태가 됩니다.

이 **부하 불균형(load imbalance)** 은 GPU 효율을 크게 떨어뜨립니다.

수천 개 스레드에 균형 있게 작업을 배분하는 것 자체가 핵심 설계 과제가 됩니다.

---

## 문제 4: 분기 발산(Branch Divergence)

교통 행동에는 조건 분기가 많습니다.

예시 규칙:

- 앞차가 느리면 차선 변경
- 신호등에서 정지
- 교차로에서 양보

코드는 보통 이런 형태가 됩니다.
```
if (light == RED)
    stop();
else
    continue();
```

GPU 스레드는 **warp** 단위로 실행됩니다.

동일 warp 내부 스레드가 서로 다른 분기를 타면, 분기 경로를 직렬화해 실행해야 합니다.

이것이 **분기 발산**이며 성능을 크게 저하시킬 수 있습니다.

교통 시스템은 본질적으로 조건 분기가 많아 이를 완전히 피하기 어렵습니다.

---

## 문제 5: 시간 스텝 간 동기화

대부분의 교통 시뮬레이터는 **이산 시간 스텝(discrete time step)** 으로 동작합니다.

각 스텝에서:

1. 차량 상태 읽기
2. 새 상태 계산
3. 시뮬레이션 상태 갱신

하지만 차량 상호작용 때문에 다음 요구가 함께 발생합니다.

- 이웃 상태 읽기
- 공유 구조 갱신
- 도로망 전반의 일관성 유지

수천 개 GPU 스레드 사이 동기화를 유지하면 오버헤드가 커집니다.

동기화가 너무 잦아지면 GPU의 이점이 사라집니다.

---

## 더 현실적인 GPU 전략

성공적인 GPU 교통 시뮬레이터는 단순 차량-병렬 모델을 그대로 쓰지 않습니다.

대신 다음 기법을 활용합니다.

- **셀 기반 모델** (예: Cell Transmission Model)
- **차선 기반 이산화(lane-based discretization)**
- **공간 분할(spatial partitioning)**
- **2단계 업데이트(two-phase update)**

이 접근들은 의존성을 줄이고 메모리 지역성을 개선합니다.

예를 들어 **Lane Cell Transmission Model (LCTM)** 은 차량을 독립 에이전트 대신 차선 셀 점유 상태로 표현할 수 있어 GPU 실행과 훨씬 잘 맞습니다.

---

## GPU 교통 시뮬레이션에서 얻는 교훈

GPU 기반 교통 시뮬레이션을 개발해 보면 병렬 컴퓨팅에 대한 일반적인 교훈을 얻게 됩니다.

### 병렬성은 모델과 맞아야 한다

모든 알고리즘이 GPU 아키텍처에 자연스럽게 매핑되지는 않습니다.

### 메모리 접근 패턴이 중요하다

메모리 지역성이 성능을 지배하는 경우가 많습니다.

### 시뮬레이션 모델 선택이 성능을 좌우한다

적절한 추상화를 고르는 것이 확장성을 결정짓습니다.

---

## 결론

GPU 가속은 교통 시뮬레이션의 확장성을 크게 높일 수 있습니다.  
하지만 그 성능을 얻으려면 다음 요소를 함께 고려해야 합니다.

- 데이터 의존성
- 메모리 접근 패턴
- 워크로드 균형
- 동기화 동작

CPU 시뮬레이터를 GPU로 단순 이식하는 방식은 대부분 실패합니다.

대신 병렬 하드웨어의 강점에 맞게 시뮬레이션 모델 자체를 재설계해야 하는 경우가 많습니다.

**시뮬레이션 모델과 하드웨어 아키텍처의 상호작용**을 이해하는 것이 대규모 시뮬레이션 시스템 구축의 핵심입니다.
